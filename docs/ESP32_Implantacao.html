1. Pré-requisitos

Hardware

ItemEspecificação
MicrocontroladorESP32 (qualquer variante com Wi-Fi — WROOM, WROVER, S3)
DisplayLCD 16×2 I2C (endereço padrão 0x27)
TecladoMatricial 4×4 (conexão direta aos GPIOs)
Cabo USBUSB-A para USB micro/Type-C (conforme modelo ESP32)
RedeWi-Fi 2.4 GHz com acesso à internet (para modo cloud)

Software

  • VS Code com extensão PlatformIO IDE instalada
  • Driver USB do ESP32 — CP210x ou CH340 conforme placa
  • Git (para clonar o repositório)
Alternativa: É possível usar o pio via linha de comando (pip install platformio) em vez de VS Code + extensão.

2. Configuração do firmware

2.1 Clonar o repositório

git clone https://github.com/SEU_USUARIO/ControleEstoque.git
cd ControleEstoque/esp32

2.2 Criar o arquivo de configuração local

O arquivo esp32/src/config.h contém valores padrão com placeholders. Crie uma cópia local que o Git ignora automaticamente:

cp src/config.h src/config_local.h
Atenção: Edite apenas o config_local.h. Nunca salve credenciais no config.h que está versionado.

2.3 Editar config_local.h

Abra src/config_local.h e preencha os campos abaixo:

// ─── Wi-Fi ──────────────────────────────────────────────────────────────────
#define WIFI_SSID     "NomeDaSuaRede"
#define WIFI_PASSWORD "SenhaDaRede"

// ─── Servidor ───────────────────────────────────────────────────────────────
// Rede local (HTTP):  "http://192.168.1.100"
// Servidor cloud:     "https://estoque.suaempresa.com"
#define SERVER_URL    "https://estoque.suaempresa.com"

// Copie do arquivo .env do servidor (campo ESP32_API_KEY)
#define API_KEY       "cole_aqui_a_chave_do_env"

// ─── HTTPS (obrigatório para servidor cloud) ─────────────────────────────────
#define USE_HTTPS

// ─── Identificação do terminal ────────────────────────────────────────────────
// Use IDs únicos quando tiver múltiplos terminais
#define DEVICE_ID     "ESP32-TERMINAL-01"

2.4 Onde encontrar a API Key

  1. Acesse o servidor VPS via SSH
  2. Execute: grep ESP32_API_KEY /opt/inventcontrol/.env
  3. Copie o valor após o = para o campo API_KEY no firmware

2.5 HTTPS e certificado Let's Encrypt

Quando USE_HTTPS está definido, o firmware valida o certificado do servidor usando o CA raiz ISRG Root X1 (Let's Encrypt), que já vem embutido no código em api_client.cpp.

O certificado ISRG Root X1 é válido até 2035. Se Let's Encrypt mudar o CA raiz antes disso, atualize o firmware via OTA.

Para rede local sem HTTPS, comente a linha #define USE_HTTPS e use SERVER_URL com http://.

3. Compilar e gravar

3.1 Via VS Code + PlatformIO

1

Abra a pasta esp32/ no VS Code (File → Open Folder). O PlatformIO detecta o platformio.ini automaticamente.

2

Conecte o ESP32 via USB. Verifique a porta COM/ttyUSB na barra de status inferior do VS Code.

3

Clique em → Upload (ícone de seta na barra inferior) ou use Ctrl+Alt+U. O PlatformIO compila e grava automaticamente.

3.2 Via linha de comando

cd esp32
pio run -t upload              # compila + grava via USB
pio device monitor --baud 115200  # monitor serial

3.3 Porta USB não encontrada

  • Windows: Instale o driver CP210x ou CH340 e verifique o Gerenciador de Dispositivos (COMx)
  • Linux: Adicione seu usuário ao grupo dialout: sudo usermod -aG dialout $USER
  • macOS: Instale o driver via Homebrew: brew install --cask silicon-labs-vcp-driver

4. Verificação após a gravação

4.1 Monitor serial

Após gravar, abra o monitor serial a 115200 baud. Você deve ver:

Conectando WiFi...
WiFi OK — IP: 192.168.1.xxx
NTP sincronizado
Verificando OTA... sem atualização
Carregando operadores: 3
Carregando produtos: 12
Sistema pronto.

4.2 Display LCD

O LCD deve exibir a tela inicial após o boot:

InventControl Selecione op.

4.3 Teste de movimento

  1. No LCD, selecione um operador, produto e quantidade
  2. Confirme o movimento
  3. No painel web (https://seudominio/reports), o movimento deve aparecer em até 5 segundos

4.4 Verificar HTTPS no monitor serial

Com USE_HTTPS ativo, você verá:

POST /api/v1/movements → 201 (HTTPS OK)

Se aparecer HTTPS FAIL ou -1, veja a seção Troubleshooting.

5. Atualização OTA (Over-the-Air)

Como funciona

No boot, o ESP32 consulta /api/v1/ota/version. Se a versão do servidor for maior que FIRMWARE_VER definida no firmware, ele baixa e grava automaticamente o novo firmware.

Publicar nova versão

  1. Incremente FIRMWARE_VER em config_local.h (ex: "1.0.1")
  2. Compile: pio run
  3. Faça upload do .pio/build/esp32dev/firmware.bin pelo painel web:
    Admin → OTA → Enviar firmware
  4. No próximo boot (ou reinício manual do ESP32), o update acontece automaticamente

OTA forçado via PlatformIO

Para gravar diretamente via rede (sem USB):

pio run -t upload --upload-port 192.168.1.XXX

O ESP32 deve estar na mesma rede. O IP aparece no monitor serial na linha WiFi OK — IP: ....

6. Troubleshooting

SintomaCausa provávelSolução
LCD não acende Endereço I2C errado ou contraste Ajuste o trimpot no módulo I2C. Teste com endereço 0x3F se 0x27 não funcionar.
WiFi não conecta SSID/senha errado, ou rede 5 GHz ESP32 suporta apenas 2.4 GHz. Verifique credenciais e aproxime da rota.
NTP não sincroniza Firewall bloqueando UDP 123 Libere a porta 123/UDP no roteador ou firewall corporativo.
HTTPS FAIL / erro -1 CA cert inválido ou expirado Verifique se o certificado do servidor está válido: curl -I https://seudominio/health. Atualize o CA cert em api_client.cpp se necessário.
API retorna 401 API Key incorreta Confira API_KEY no firmware vs ESP32_API_KEY no .env do servidor.
API retorna 429 Rate limit do nginx atingido Aguarde 1 minuto. Se persistir, o terminal está enviando requisições em loop — verifique o código.
Operadores não carregam Servidor indisponível ou URL errada Verifique SERVER_URL e se o servidor está acessível pelo IP/rede do ESP32.
OTA falha Firmware maior que 50% da flash O ESP32 precisa de partições OTA ativas no platformio.ini (board_build.partitions = min_spiffs.csv).

Comando de diagnóstico rápido

No servidor VPS, verifique se o ESP32 está chegando:

tail -f /opt/inventcontrol/logs/movements/movements.log | python3 -m json.tool
